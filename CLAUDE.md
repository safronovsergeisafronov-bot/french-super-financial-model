## Как выбирать методологию — автоматические правила

Читай запрос → определи тип задачи → применяй соответствующую методологию (или комбинацию).

```
Тип задачи                                  Методология
─────────────────────────────────────────────────────────────────
Изменение формул / расчётов                 TDD + Property-Based
Новая бизнес-функция (новый продукт,        BDD → TDD
  новый тип расхода, новый показатель)
Рефакторинг существующего кода              TDD (сначала тесты,
                                             потом переписываешь)
Исправление бага в расчётах                 TDD (воспроизведи баг
                                             тестом, потом фикс)
UI изменения без логики (цвет, размер,      Self-Reflection только
  текст, layout)
Крупная новая секция (>2 компонента,        SDLC-агенты → BDD → TDD
  новая страница)
Интеграция внешнего сервиса / API           SDLC-агенты → TDD
Объяснение / анализ кода                    Self-Reflection только
```

**Правило:** Self-Reflection применяется ВСЕГДА в конце, независимо от других методологий.

## Методологии

### TDD (Test-Driven Development)

**Применяется:** изменение расчётов, баги в логике, рефакторинг.

**Порядок:**
1. Напиши тест в `__tests__/` — ДО кода
2. Убедись что тест падает (red)
3. Напиши минимальный код чтобы тест прошёл (green)
4. Рефактори без поломки теста

**Правило:** вся математика живёт в `lib/calculations.js`, не в компоненте.

```bash
npm test             # запустить тесты
npm run test:watch   # watch mode при разработке
```

---

### BDD (Behavior-Driven Development)

**Применяется:** новая бизнес-логика, новый продукт/поток.

**Порядок:**
1. Прочитай `specs/` — есть ли уже сценарий?
2. Если нет — создай `.feature` файл (Дано/Когда/Тогда) до кода
3. Реализуй код строго по сценарию
4. Перейди к TDD для покрытия тестами

**Шаблон:**
```gherkin
Функционал: [название]
  Сценарий: [описание]
    Дано [начальное состояние]
    Когда [действие пользователя или системы]
    Тогда [ожидаемый результат с конкретными числами]
```

---

### SDLC-агенты (для крупных задач)

**Применяется:** новая страница, интеграция API, крупный рефакторинг (>2 файла).

**Порядок ролей — явно обозначай каждую в ответе:**
1. **[АРХИТЕКТОР]** — структура данных, интерфейсы, схема компонентов
2. **[КОДЕР]** — реализация по архитектуре
3. **[ТЕСТИРОВЩИК]** — тесты, попытка сломать, граничные случаи

---

### Property-Based Testing

**Применяется:** всегда при изменении формул в `lib/calculations.js`.

**Инварианты — обновляй `__tests__/invariants.test.js` если добавляешь новые поля:**
- `netProfit = totalRevenue - totalCost` (всегда)
- `totalRevenue >= 0` при любых ползунках
- `isProfitable === (netProfit >= 0)` (флаг и число всегда согласованы)
- `totalCost = managerCostTotal + teacherCostTotal + otherCosts`
- `breakeven` — целое положительное число

---

### Self-Reflection (всегда, в конце)

**Перед тем как объявить задачу завершённой, пройди мысленно:**
- [ ] Граничные значения не ломают расчёты? (0 уроков, 0 менеджеров, деление на 0)
- [ ] `npm test` — все тесты проходят?
- [ ] Мобильный layout не сломан?
- [ ] Новые переменные имеют понятные имена (не `lr`, `tc`, `mb`)?
- [ ] Не добавил ничего лишнего что не просили?
