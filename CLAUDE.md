# CLAUDE.md — Правила работы для Claude Code

Этот файл загружается автоматически. Следуй этим правилам **без напоминаний**.

---

## Проект

**French.Super** — интерактивная финансовая модель онлайн-школы французского.
Стек: Next.js 16, React 19, App Router, деплой на Vercel.

---

## Методологии (применяй автоматически)

### 1. TDD — для любой бизнес-логики

**Когда:** любое изменение в расчётах (выручка, маржа, ROI, безубыточность).

**Порядок:**
1. Напиши тест в `__tests__/` — сначала, до кода
2. Убедись что тест падает (red)
3. Напиши минимальный код чтобы тест прошёл (green)
4. Рефактори (refactor)

**Команды:**
```bash
npm test          # запустить тесты
npm test -- --watch  # watch mode
```

**Правило:** функции расчётов (`calculateModel`, и т.д.) должны быть вынесены из компонентов в отдельные файлы `lib/` и покрыты тестами.

---

### 2. BDD — для новых фич и бизнес-сценариев

**Когда:** добавляешь новый блок, поток, или бизнес-логику.

**Порядок:**
1. Прочитай соответствующий `.feature` файл в `specs/`
2. Если файла нет — создай его в формате Gherkin (Дано/Когда/Тогда)
3. Реализуй код строго по сценарию

**Шаблон:**
```gherkin
Функционал: [название]
  Сценарий: [описание]
    Дано [начальное состояние]
    Когда [действие]
    Тогда [ожидаемый результат]
```

---

### 3. Self-Reflection — перед каждым финальным ответом

**Когда:** всегда, перед тем как объявить работу завершённой.

**Чек-лист (пройди мысленно):**
- [ ] Нет ли багов в граничных значениях? (0 уроков, 0 менеджеров, деление на 0)
- [ ] Сохранена ли логика безубыточности?
- [ ] Не сломан ли мобильный layout?
- [ ] Все расчёты детерминированы (нет случайности)?
- [ ] Тесты написаны и проходят?

---

### 4. SDLC-агенты — для крупных задач (>2 компонента)

**Когда:** добавляешь большой блок (новый раздел, графики, сценарии роста).

**Порядок ролей:**
1. **Архитектор** — опиши структуру данных и интерфейсы
2. **Кодер** — реализуй по архитектуре
3. **Тестировщик** — напиши тесты и попытайся сломать

Явно обозначай каждую роль в своём ответе.

---

### 5. Property-Based Testing — для финансовых инвариантов

**Когда:** меняешь формулы расчётов.

**Инварианты которые ВСЕГДА должны быть истинны:**
- `net = rev - total_cost` (всегда)
- `rev >= 0` при любых ползунках
- `margin = (profit / rev) * 100` если `rev > 0`, иначе `0`
- ROI > 0 если `manager_cost > 0`
- Безубыточность (`be`) → положительное целое число

Добавь тесты для этих инвариантов в `__tests__/invariants.test.js`.

## Структура проекта

```
french-super-financial-model/
├── CLAUDE.md                      # этот файл
├── app/
│   ├── layout.jsx
│   └── page.jsx
├── lib/
│   └── calculations.js            # вся бизнес-логика (без React)
├── specs/
│   └── *.feature                  # BDD сценарии
├── __tests__/
│   ├── calculations.test.js       # TDD тесты логики
│   └── invariants.test.js         # Property-based инварианты
└── docs/
    └── methodologies.md           # подробное описание подходов
```

**Правило:** бизнес-логика должна жить в `lib/calculations.js`, а не внутри React-компонента. Компонент только отображает.

## Стиль кода

- Никаких `any` типов если перейдём на TypeScript
- Имена переменных — понятные (не `lr`, `pt`, `mk` — только в тестах допустимо)
- Комментарии только там где формула неочевидна
- Не добавляй фичи которые не просили
